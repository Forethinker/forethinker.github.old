My attempt to replace Bash with Fish

August 11, 2013 [ hack ] #shell, #cygwin Read in 5 min.

I was quite happy with my Bash setup, until someone took my blindfold away and revealed to me the Fish shell. So I went on to check if there is something better. Well, now I know what zsh (1990) offers and that fish shell is the newest shell (2005). If you use Mac or Ubuntu, I encourage you to look at Hacker Codex, which has been the inspiration for this article.

I masochistically installed it on my Cygwin. Here is how it went. UPDATE: The conclusion is that fish is not mature enough for me to use it full time, but its features are very promising and I want people to give it a try... maybe go further than where I have been.

Installation

cd /opt
git clone https://github.com/fish-shell/fish-shell/
automake && ./config
vim fallback.h
# remove the following line start on line 89

##ifdef TPARM_SOLARIS_KLUDGE

#/**
#   Solaris tparm has a set fixed of paramters in it's curses implementation,
#   work around this here.
#*/

##define tparm tparm_solaris_kludge
#char *tparm_solaris_kludge(char *str, ...);

finally install

make && make install

Setup "fish.config" == ".bashrc"

I changed my Cygwin.bat to be:

@echo off

C:
chdir C:\cygwin\usr\local\bin
fish

and also edit /etc/passwd and change my shell to /usr/local/bin/fish

Then I opened my mintty console. The color of vim looks aweful in Cygwin or cmd.exe because I do not think they support 256 color. Well I get the following message:

fish: Could not connect to universal variable server, already tried manual restart (or no command supplied). You will not be able to share variable values between fish sessions. Is fish properly installed?
Welcome to fish, the friendly interactive shell
Type help for instructions on how to use fish

Well looking at the document, there is daemon called fishd that is supposed to do its job. Uh... it is not in my PATH? There is one in /opt/fish-shell just symlink it. Then you need to make it run when you launch your shell. I have the following lines in my `~/.config/fish/config.fish

set PATH /usr/local/bin $PATH
set PATH /usr/sbin $PATH
fishd
set -g BROWSER firefox.exe
cd

This is the very minimum, I think, to get fish working properly. While getting fish to work with screen, I kept on getting this message again. Anyway, I had to replace alias to function when I ported my '.bashrc' so that I wouldn't get any warning. I accomplished this with the following vim command:

'<,'>s/alias \(\w\{-}\)='\(.*\)'/function \1\r\t\2\rend\r/g

My home directory was set to System32, despite setting the home directory on /etc/passwd. My guess is that it is because fish shell is called by cmd.exe, because nothing showed up on a profile dump. Well, hence cd in the last code snippet.

Fixing LS

The problem is mentioned in the github. In short you get error messages. So I just gutted (commented) the following lines from '/usr/local/share/fish/functions/ls.fish'

if not set -q LS_COLORS
    if type -f dircolors >/dev/null
        eval (dircolors -c)
    end
end

and it went away. I believe the problem is with discolor, which I do not think is needed.

On a side note, make sure you do not define function ls anywhere. It thinks it is going to get into an infinite loop. To me, this does not seem very robust, as alias is a syntactic sugar for function. I had to modify my ls to exlucde ntuser files which clutters up home directory unnecessarily whenever I ls oh and I had to escape the kleene star *

set param $param --indicator-style=classify --time-style=iso --ignore=ntuser\*

Vim

the directory in PATH is in different order than Cygwin, so I had to add the following line to make sure that gVim is not invoked instead of vim. Otherwise, vim will hang because the shell does not know what to do with the GUI version.

function vim
    /bin/vim $argv
end

GNU Screen

I added the following line so that fish will run whenever I create a new window

defshell /usr/local/bin/fish

Syntax

Okay, consult the man page with help, but just know that the command substitution works with just parenthesis. In another word without '$'

Bunch of message with x86

You will get the following message whenever you try to auto-complete a command. 

fish: The file 'x86' is not executable by this user
in command substitution
    called on standard input,

The problem is hard coded in the source code and directories with parenthesis are not being parsed correctly. I posted the issue on Github.

Other problems

Fish does not seem to be snappy at times like Bash. Maybe it has to do with "spawning new thread problem" of Cygwin and it would probably has to do with caching too. For me, hanging shell is a deal breaker. The very reason why I go command line is the speed. Yes it is sad, but I just cannot give up my workflow.
