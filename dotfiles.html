Dotfiles management

September 27, 2013 [ hack ] #shell Read in 3 min.

I guess the conventional way to control dotfile is using github. It is not a bad idea, but I stick with Dropbox for versioning. Yes, it is a version control for non-power user, but it is pretty handy to have your .vimrc every time you edit it. I typically document my dotfiles in the code and I have not found the commmit message to be useful for dotfiles.

Windows

Symlink dotfiles using Link Shell Extension. This allows me to edit dotfiles in either Dropbox or in my profile without worry about copying them over. I found hardlinks to not work as I expect from Unix environment. It is much simpler to use symlinks than trying to keep dotfiles in Dropbox and linking all your applications to read from them. Also expecting your dotfiles to exist in your home directory is a good practice in case you are in an environment where Dropbox is not available (ARM server comes to my mind).

Remote (Linux)

The only time I need to copy dotfiles to another server is when I log in to the server for the first time. I need to copy my public keys and dotifles and install some essential programs. First I save the host name in the .ssh/config file and then I do everything I need with the following function:

sshk() {
    ssh-copy-id -i ~/.ssh/id_rsa.pub "$@"
    scp -q $HOME/Dropbox/dev/dotfiles/.??* "$@":
    ssh "$@" 'sudo apt-git install git vim; git clone https://github.com/robbyrussell/oh-my-zsh .oh-my-zsh; git clone https://github.com/gmarik/vundle.git ~/.vim/bundle/vundle; vim +BundleInstall +qall'
}

The last line clones Oh-My-Zsh and Vundle and then installs the plugins. Just save this in .bashrc and then run it with sshk hostName

It is exhilarating that you can feel home with just a few key strokes.
